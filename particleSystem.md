# 流体シミュレーションとパーティクルシステムの連携

立派な流体シミュレーションとその描画するためレイマーチングシェーダーがあってもデータの操作も考えないと初期のステートから動きません。
シミュレーションだけではなく、面白いエフェクトを再現できるためのデータ入力システムも必要です。

もちろんこれは新しい問題ではありません、今までの立派な研究者たちが何をみるべきですね。

まず数学の世界、特に天気の研究にはリアルの現在のデータを使ってこれからどう変更を知りたいところです。
それは、「初期条件」を決めて、それからデータをいじらず観察のみが多いと思いますね。

数学敵にこれはすごく自然ですし、すごくきれいなやりかたですが、ゲームの場合はどうでしょうかね？
実は大きく3つのデメリットがあります：

1. 面白いエフェクトの初期条件を探すのは非常に難しい
2. 面白いエフェクトができても、調整が非常に難しい
3. プレイヤーとの連動がまったくない

簡単なシステムだと初期条件のみできれいなエフェクトができます、通常のパーティクルシステムが大体それです。
でも流体の場合はとっても要素しづらい、初期条件をちょっとだけいじったら大分違う結果になる場合がかなりあります。

これを解決するために、空間の一部だけを集中して、一部だけの値をいじって感じにします。
データを出すイメージですのでエミッターっと呼びます。

## エミッター
コンセプトがかなり簡単と思います。
区域の情報と設定したいデータをセットにして、 GPU に送ります。
それで各ボクセルがその区域の中かどうか確認して、はいてるであればデータを上書きすればいいでしょうね。

今のところはエミッター3個くらいまでしかないので処理は深く考えなくても全然平気です。

```hlsl
for (uint i = 0; i < EmitterCount; ++i)
{
    Emitter em = Emitters[i];

    if (insideEmitter(em, cellWorldPos))
    {
        advectedSmokeProps.r = max(advectedSmokeProps.r, em.heat);
        advectedSmokeProps.g = max(advectedSmokeProps.g, em.density);
        advectedSmokeProps.b = max(advectedSmokeProps.b, em.fuel);

        advectedVel.xyz = em.velocity;
    }
}
```

1件だけすごく気を付けないといけない所があります：データ配置。
GPUとCPUを共有する struct は必ず16の段バイトにしないといけません。
そのためにパディング用の変数を追加する必要です！
```
struct Particle
{
    float3 position; // 12 bytes
    float radius;    // 4  bytes
    float3 velocity; // 12 bytes
    float heat;      // 4  bytes
    float density;   // 4  bytes
    float fuel;      // 4  bytes

    float2 padding;  // 8  bytes

    // データ 40 bytes + パディング 8 bytes = 48 bytes = 3 * 16 bytes
};
```

## 気圧のエミッター
ある区域に気圧をあげて、その中の流体を外に出す機能が普通にほしいですが、直接気圧を上書きするのはかなり危険です。
質量保存の法則みたいな条件いろいろ守るために慎重に気圧を計算していますね。
その上に無理やり気圧のデータを上書きすると、シミュレーションがかなり不安定になります。

でも希望があります、いい感じのトリックがあります。
気圧の計算が発送が0の状態を目指しています。
発送が高かったら、キャンセルするために気圧が低くなります。
反対も同じで、発送が低い所でキャンセルするために気圧が上がります。

ので気圧を操作するのは、気圧ではなく発送をいじればいいです！

```hlsl
for (uint i = 0; i < EmitterCount; ++i)
{
    Emitter em = Emitters[i];

    if (insideEmitter(em, worldPos))
    {
        div -= em.expansion;
    }
}
```

## パーティクルエミッター
エミッターのシステム際あれば、流体シミュレーションを操作できますが、新しい問題があります。
今度はいい感じにエミッターを操作する必要になりました。。

エミッターの大きさや操作するデータなどの全部を時間に操作したいですね。
ですが、どんなゲームエンジンでもこんな感じにデータを管理できるシステムがすでにあります。
パーティクルです。

これもコンセプトが簡単です。
パーティクルがエミッタ役してもらいます。
必要なデータだけパーティクルから受け取って、エミッターに対してGPU側に渡します。

大きいな違いは数ですね。。。　簡単に数百パーティクルが追加できますのでその程度の数を対応しないとですね。
今までの単純のループは全ボクセルで計算していますのでかなり重くなります。

## ラスタライズ処理
解決するのはパーティクルをラスタライズをします
今まではボクセルごとに何のパーティクルが触れているかを確認していますが、その反対が圧倒的に早い。

まず各パーティクルが何のボクセルが触れているかを確認して、その情報がテクスチャに保存します。
ボクセルの処理の時はそのテクスチャを参考してデータを操作します

このプロセスが描画するとまったく同じです、のでラスタライズっと呼びます。

これも1件だけ気を付けないといけない所があります。
GPU がもちろん非同期で動いていますので、複数のパーティクルが同時に処理します。
でも2つのパーティクルが同じボクセルの情報を上書きしたいケースが十分あります。

しっかりアトミック操作でデータを書けば問題ないです。私の実装で `InterlockedMax()` を使いました。
