# 流体シミュレーションとパーティクルシステムの連携

立派な流体シミュレーションと、それを描画するためのレイマーチングシェーダーがあっても、データの操作を考えないと初期状態から動きません。
シミュレーションを行うだけでなく、面白いエフェクトを再現するためのデータ入力システムも必要になります。

まず数学の世界、特に気象などの研究では、現実の現在のデータを使ってこれからどう変化するかを知りたいですよね。そういった場合は、「初期条件」を決めたら、その後はデータをいじらずに観察のみを行うことが多いと思います。

数学的にはこれはすごく自然ですが、ゲームの場合はどうでしょうか？ 実は大きく3つのデメリットがあります：

1. 面白いエフェクトになる初期条件を探すのは非常に難しい
2. 面白いエフェクトができても、調整が非常に難しい
3. プレイヤーとの連動がまったくない

簡単なシステムなら初期条件のみできれいなエフェクトが作れます。通常のパーティクルシステムは大体それですね。
でも流体の場合はとっても予想しづらく、初期条件をちょっといじっただけで結果が大きく変わってしまうことがかなりあります。

これを解決するために、空間の一部だけに集中して、そこだけの値を継続的にいじるような仕組みにします。
データを放出するイメージなので「エミッター」と呼びます。

## エミッター
コンセプトはかなり簡単だと思います。
影響範囲の情報と設定したいデータをセットにしてGPUに送ります。そして各ボクセルがその区域の中に入っているか確認し、入っていればデータを上書きすればいいわけです。

今のところはエミッターが3個くらいまでしかないので、処理の重さは深く考えなくても全然平気です。

```hlsl
for (uint i = 0; i < EmitterCount; ++i)
{
    Emitter em = Emitters[i];

    if (insideEmitter(em, cellWorldPos))
    {
        advectedSmokeProps.r = max(advectedSmokeProps.r, em.heat);
        advectedSmokeProps.g = max(advectedSmokeProps.g, em.density);
        advectedSmokeProps.b = max(advectedSmokeProps.b, em.fuel);

        advectedVel.xyz = em.velocity;
    }
}
```

ただ、1点だけすごく気を付けないといけないところがあります。それはデータ配置（アライメント）です。
GPUとCPUで共有する `struct` のサイズは、必ず16バイトの倍数にしないといけません。
そのためにパディング用の変数を追加する必要があります！

```hlsl
struct Particle
{
    float3 position; // 12 bytes
    float radius;    // 4  bytes
    float3 velocity; // 12 bytes
    float heat;      // 4  bytes
    float density;   // 4  bytes
    float fuel;      // 4  bytes

    float2 padding;  // 8  bytes

    // データ 40 bytes + パディング 8 bytes = 48 bytes = 3 * 16 bytes
};
```

## 圧力のエミッター
ある区域の圧力を上げて、その中の流体を外に押し出す機能が普通は欲しいところですが、圧力を直接上書きするのはかなり危険です。
質量保存の法則などの条件を守るために、すでに慎重に圧力を計算していますからね。その上に無理やり圧力のデータを上書きすると、シミュレーションがかなり不安定になってしまいます。

でも希望はあります。いい感じのトリックがあるんです。
圧力の計算は、本来「発散（Divergence）が0の状態」を目指して行われています。発散が高い（湧き出しがある）と、それを打ち消すために圧力が働きます。
ですので、圧力を操作したいなら、圧力そのものではなく「発散」をいじればいいのです！

```hlsl
for (uint i = 0; i < EmitterCount; ++i)
{
    Emitter em = Emitters[i];

    if (insideEmitter(em, worldPos))
    {
        div -= em.expansion;
    }
}
```

## パーティクルエミッター
エミッターのシステムさえあれば流体シミュレーションを操作できますが、ここで新しい問題が生まれます。
今度は「エミッターをいい感じに操作する」必要が出てきました。

エミッターの大きさや設定するデータなどを、すべて時間経過に合わせて操作したいですよね。
実は、どんなゲームエンジンでも、こんな風にデータを管理できるシステムがすでに存在します。
そう、パーティクルです。

これもコンセプトは簡単です。
パーティクルにエミッター役をやってもらいます。必要なデータだけをパーティクルから受け取って、エミッター情報としてGPU側に渡すだけです。

ただ、大きな違いはその「数」ですね。。簡単に数百個のパーティクルが追加されてしまうので、それだけの数に対応しないといけません。
今までの単純なループでは、全ボクセルに対して各エミッターの判定を計算していたので、数が数百になるとかなり重くなってしまいます。

## ラスタライズ処理
この問題を解決するために、パーティクルをラスタライズします。
今までは「各ボクセルがどのエミッター（パーティクル）に触れているか」を確認していましたが、その逆のほうが圧倒的に速いのです。

まず「各パーティクルがどのボクセルに触れているか」を確認し、その情報をテクスチャに書き込みます。
そしてボクセルの処理の時は、そのテクスチャを参照してデータを操作するようにします。

このプロセスは、画面にポリゴンを描画する仕組みとまったく同じなので「ラスタライズ」と呼びます。

これも1点だけ気を付けないといけないところがあります。
GPUはもちろん非同期で動いているので、複数のパーティクルが同時に処理されます。
そのため、2つのパーティクルが「同じボクセルの情報を同時に上書きしようとする」ケースが十分あり得ます。

これは、しっかりとアトミック操作でデータを書き込めば問題ありません。私の実装では `InterlockedMax()` などの関数を使いました。
