# 流体シミュレーションの基本

詳しく書きすぎると一冊の本になってしまいますので、ここではざっくりと解説します。

基本的な仕組みとして、まずは空間をグリッドに分割します。この分割された各セルを「ボクセル」と呼びます。
ボクセルごとに、その場所にある流体の状態の値を保存していきます。具体的には、速度や煙の密度、温度などのデータですね。
実際の実装では、これらを3次元テクスチャ）に保存して管理します。

そして、このデータに流体力学の魔法をかけることで、流体がどう動くかを計算していきます。
シミュレーションは、主に以下の3つのステップを繰り返すことで成り立っています。

1. 外部の力（External Forces）
2. 圧力（Pressure）
3. 移流（Advection）

1つずつ見ていきましょう。

## 外部の力
ここが一番分かりやすいですね。ゲームでお馴染みの定番の物理演算と同じように、加速度の値を速度に加算するだけです。

炎や爆発のエフェクトを描く上で、必要な力は大体以下の2つだけです。

* **重力**: 煙の「密度」が高いほど、下方向へ強く加速する力
* **浮力**: 煙の「温度」が高いほど、上方向へ強く加速する力

この2つは、以下のように同時に簡単に計算できます。

```hlsl
float buoyancyForce = (heat * Buoyancy) - (smokeDensity * SmokeWeight);
vel.y += buoyancyForce * DeltaTime;
```

物理シミュレーションとしてはこれだけでも動きますが、見た目をさらにいい感じにするために、物理的な根拠は薄いものの、エフェクト表現として重要な2つの力を追加します。

### カールノイズ
これは「環境風」の役割になります。
現実世界には、完全に静止した空気というものは存在しません。常にどこかがちょっとずつ揺らいでいます。その現象を再現するためにノイズを加算します。

ただし、ここで1つだけ気を付けなければいけない点があります。
それは、流体が勝手に圧縮されたり消滅したりしないように、湧き出しや吸い込みがない「発散が0のノイズ」を使う必要があるということです。

それを解決してくれるのが「カールノイズ」です。
数学的に、あるベクトル場の回転を計算したものの発散は必ず0になるという性質があります。そのため、普通のシンプレックスノイズなどの回転を計算してあげることで、発散が0の理想的なノイズ場が作れます。

このノイズの値を、普通の力と同じように速度に加算してあげればOKです。

### ボルティシティ・コンファイメント（渦度閉じ込め）
これはかなり細かい、こだわりの部分です。
現実世界に比べたらシミュレーションには、大きな違いが1つあります。 それは、シミュレーション内だと回転がどんどん弱くなってしまうということです。
これは流体の計算式が間違っているわけではなく、時間を不連続に区切って計算しているせいです。
どんなシミュレーションでも時間は1フレームずつ進みます。どれだけ頑張って時間の間隔を細かくしても、不連続であるという事実は変わりません。その計算誤差のせいで、渦のエネルギーが自然と消えていってしまうのです。

失われてしまうなら、人工的に「回転の力を補正して加算してあげる」必要があります。それがこのボルティシティ・コンファイメントです。

## 圧力の計算
今回は非圧縮性のシミュレーションを行っているので、圧力の役割は発散をゼロに抑え込む抵抗力になります。
問題は、この圧力の計算式（ポアソン方程式）には一発で答えが出る解析解が存在しないということです。地道に数値計算で近似解を探すしかありません。

そのせいで、実装を工夫しないとこのステップの処理がめちゃくちゃ重くなってしまいます。

### ヤコビ法
一番定番でシンプルな計算方法です。
現在の圧力の値を魔法の数式に入力すると、ちょっとだけ正確になった新しい圧力の値が出てきます。

```hlsl
float dx = BoundsSize.x / float(Resolution.x);
float div = DivergenceWrite[id];
PressureWrite[id] = (pL + pR + pD + pU + pB + pF - (dx * dx * div)) / 6.0;
```

数式の詳細が分からなくてもコード自体は書けますが、この手法の弱点は知っておくべきです。
ヤコビ法は隣のボクセルの情報しか参照しません。そのため、気圧の変化が1ステップごとに1ボクセルずつしか伝播しないのです。つまり、グリッド全体のサイズと同じくらいの回数（何十回、何百回と）ループを繰り返さないと、全体に圧力が伝わりきりません。

### マルチグリッド法
隣のボクセルにしか情報を伝播できないヤコビ法の弱点を、スマートに解決するのが「マルチグリッド法」です。

グリッド（ボクセル）のサイズが大きいほど、情報が遠くまで早く伝わりますが、データの精度は低くなります。
逆に、グリッドが小さいほど、情報の伝播は遅くなりますが、データの精度は高くなります。
この性質を利用して、いろんなサイズのグリッドを組み合わせて計算を高速化するのがマルチグリッド法です。「伝播の早さ」のために粗いグリッドを使い、「精度の高さ」のために細かいグリッドを使います。

処理の流れは、基本的にアルファベットの「V」の形に進みます。

* 一番細かいグリッドで、数回だけヤコビ法を実行して圧力をなじませる。
* ダウンサンプリングして1段階粗いグリッドに移動し、また数回ヤコビ法を使う。
* これを繰り返して、一番粗いグリッドまでデータを落とし込む（ここで一気に圧力が全体に伝播する）。
* 今度は逆にアップサンプリングして、グリッドを徐々に細かくしていき、その都度ヤコビ法で精度を補正する。
* 一番細かいグリッドまで戻ってきたら、計算終了。

これで、普通にヤコビ法を何百回も回すより圧倒的に早く正確な圧力が求まります。

## 移流の計算
簡単にいうと、「ボクセルの中身の値を、現在の速度の方向へ移動させる」という処理です。
中身がどこへ移動するかを愚直に計算するのは難しくありませんが、問題は「移動した結果、どのボクセルに収まるか」です。ぴったり綺麗に隣のボクセルの中央に移動してくれることなんて、まずあり得ません。大体はボクセルとボクセルの半端な境界線上に着地します。

ですが、考え方をちょっとだけ変えると簡単に解決できます。
「現在のボクセルの中身がどこへ行くか」ではなく、「現在のボクセルが**どこから来たのか**」を逆算するのです。
逆算した元いた場所がボクセルの隙間の半端な位置だったとしても問題ありません。3Dテクスチャの機能であるトリリニアサンプリングを使えば、周囲のボクセルから補間された値を簡単に取得できます。

### マコーマック法（MacCormack Method）
これは、簡単な例で説明したほうが分かりやすいと思います。
初心者がよく書く定番の物理演算コードは、大体こんな感じですよね。

```csharp
vel += acc * deltaTime;
pos += vel * deltaTime;
```

実はこれ、物理学的には厳密には不正解です。問題は何でしょう？
ボルティシティ・コンファイメントのところと同じで、時間を不連続に計算しているせいです。

単純なケースなら正確な計算式を作れますが、複雑な流体の場合は完全に正確な計算は不可能です。
（ちなみに、もし加速度 `acc` が完全に一定であるなら、正解は `pos += vel * deltaTime - acc * deltaTime * deltaTime / 2;` になります）

「マコーマック法」は、非常に賢いアプローチでこの時間離散化による計算誤差を減らし、より正確な値に近づける手法です。
完全ではないですが、通常のシンプルな移流に比べて、格段にシャープで良い結果が得られます。
仕組みを簡単にすると、以下のように計算しています。

1. 現在の速度を使って、1ステップ後ろへ移流させてみる（仮の過去位置を求める）。
2. その移動先から、今度は1ステップ前へ移流させてみる。
3. 往復して戻ってきた位置と、最初のスタート位置を比べる。この2つの位置のズレが、離散化によって生じた「エラーの2倍」の量になります（往復で2回移流させているため）。
4. 最初に求めた1ステップ前の値から、このエラーの半分を差し引いて補正する。これによって、理論上の正確な値にグッと近づきます。

```hlsl
// MacCormack Advection
float3 backStep = uvwVel * DeltaTime * SimScale;
float3 backUvw = uvw - backStep;
float4 backVel = VelocityRead.SampleLevel(sampler_LinearClamp, backUvw, 0);
float3 backUvwVel = backVel.xyz / BoundsSize;

float3 forwardStep = backUvwVel * DeltaTime * SimScale;
float3 forwardUvw = backUvw + forwardStep;
float3 prevUvw = backUvw + (uvw - forwardUvw) / 2.0;

float4 advectedSmokeProps = SmokePropRead.SampleLevel(sampler_LinearClamp, prevUvw, 0);
float4 advectedVel = backVel;
```
